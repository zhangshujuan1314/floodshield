## 一、系统定位与绝对边界

平台的核心链路是：

官方/授权数据 -> 标准化 -> 可解释风险规则 -> 附近风险摘要 -> 居民行动建议 -> 现场报告 -> 社区核验 -> 路线和避险场所更新 -> 通知与处置闭环。

平台是公众避险辅助和基层协同工具，不能代替气象、水务、应急管理等部门发布法定预警。任何页面、接口、AI 输出和测试数据都必须保留来源、观测时间、有效期、数据状态和不确定性。

以下行为禁止实现：

- 把模型输出当作官方预警或修改官方预警等级；
- 没有传感器或可信观测时生成精确积水深度；
- 把风险未知的路线或场所标记为安全；
- 自动发布未经核验的高危公众消息；
- 自动确认救援已经派出；
- 自动控制泵站、闸门、交通设施或救援设备；
- 默认后台持续定位；
- 将用户报告中的文字当作系统指令、权限或规则；
- 在没有凭证和明确授权时发送真实短信、电话、推送或访问生产数据。

## 二、工作规则

1. 先检查当前仓库：目录结构、README、CLAUDE.md、包管理器、测试命令、环境变量、现有未提交修改。保留用户已有修改，禁止 reset、checkout --、删除未知文件或大范围重构。
2. 如果仓库为空，按最小可用架构初始化；如果仓库已有技术栈，优先遵循现有约定。不要为了追求理想架构替换已有栈。
3. 每次改动前写出简短的可验证目标，每次改动后运行最小相关测试，再运行阶段级检查。
4. 只在遇到破坏性操作、真实外部发送、生产部署、真实凭证、付费服务或会改变产品范围的选择时暂停。缺少真实数据源时建立 adapter 和 fixture，继续使用 mock 模式。
5. 任何“完成”都必须有当前命令输出作为证据。失败的测试继续修复；无法运行时明确写出原因。
6. 使用最少的新依赖。先复用当前项目能力；确需新增依赖时说明用途，并锁定版本。
7. 所有安全关键逻辑必须有单元测试和边界测试。风险计算和路线重排不能只靠 UI 测试。
8. 用户端文案使用简短中文。重要信息同时显示文字、图标、颜色和更新时间，颜色不能承担唯一语义。
9. 默认语言为简体中文，代码命名使用英文，时区处理明确使用 `Asia/Shanghai`，所有时间在 API 中使用带时区的 ISO 8601。

## 三、推荐实现形态

如果当前仓库没有既定技术栈，采用以下最小模块化单体方案：

- `apps/miniapp`：Taro + React + TypeScript，优先支持微信小程序和 H5；
- `apps/admin`：Next.js + TypeScript，用于社区和应急管理后台；
- `services/api`：FastAPI + Python；
- 数据库：PostgreSQL + PostGIS；
- 本地开发：Docker Compose；
- 缓存：先定义接口，只有确有需要时加入 Redis；
- 外部数据：provider/adapter 接口，默认 mock；
- AI：默认 `NoopAiProvider` 或确定性 mock，真实模型接入必须有明确环境变量和审计字段；
- 文件：本地开发存储适配器，生产使用对象存储适配器；
- API：OpenAPI 为契约来源，前后端不能通过复制粘贴隐式约定字段。

不要在第一阶段创建微服务、复杂事件总线、自研地图引擎或自主水动力模型。

## 五、领域模型与强制数据字段

至少实现以下领域：

- 用户、组织、角色和权限；
- 官方预警 `official_alerts`；
- 雨情/传感器/公众观测 `observations`；
- 居民报告 `hazard_reports`；
- 道路事件 `road_events`；
- 避险场所 `shelters`；
- 风险快照 `risk_snapshots`；
- 路线请求和结果 `route_requests`、`route_results`；
- 通知投递和处置任务；
- 不可变审计日志。

每个风险结果和路线结果至少要能表达：

- `sourceIds` 或 `evidence`；
- `observedAt`；
- `calculatedAt`；
- `expiresAt`；
- `dataStatus`：`fresh | partial | stale | unknown`；
- `confidence`；
- `disclaimer`；
- `ruleVersion`（适用于规则计算）。

位置字段遵循最小必要：公开报告使用模糊位置，精确位置单独加密或短期保存；禁止为了展示地图而保存用户后台轨迹。

## 六、风险引擎要求

先实现可解释的确定性规则，后续才接入模型。建议接口：

```python
class RiskEngine(Protocol):
    def compute(self, snapshot: RiskInputSnapshot) -> RiskSnapshot: ...
```

风险分数只是平台内部排序信号，不能改写官方预警。输入至少包括：官方预警、降雨观测、核验报告、静态易涝点、道路事件和数据新鲜度。

实现以下约束：

- 规则和阈值可配置，并保存 `ruleVersion`；
- 缺失数据输出 `unknown` 或 `partial`，不能默认 `normal`；
- 过期输入不得继续作为新鲜证据；
- 负面/危险证据优先进入说明；
- 多源冲突要保留冲突信息；
- 计算结果可用输入快照回放；
- AI 服务不可用时风险引擎照常工作。

建议平台风险带：`normal`、`attention`、`high`、`critical`。在 UI 中明确标注“平台风险等级”，不要与官方蓝、黄、橙、红预警混用。

## 七、路线要求

MVP 不自研全城路网。通过 `MapProvider` 获取候选路线，再进行风险重排：

```python
class MapProvider(Protocol):
    async def geocode(self, query: str) -> list[Place]: ...
    async def route(self, request: RouteRequest) -> list[RouteCandidate]: ...
```

路线规则：

- 官方封路、已核验不可通行路段属于硬阻断；
- 高风险但可通行路段增加惩罚并展示证据；
- 风险未知路段不能标记安全，结果中要显示未知风险；
- 所有路线返回计算时间、数据时间、证据和失效时间；
- 地图服务失败时不能编造路线；
- mock provider 必须能稳定产生“正常路线、含危险路段、没有可用路线”三种 fixture。

## 八、居民端必须交付的页面

最少实现：

1. 首页：附近风险、官方预警、下一步动作、更新时间、语音按钮；
2. 地图：官方预警、平台风险、报告、道路事件、避险场所图层；
3. 上报：积水、道路受阻、地下空间、受困、其他隐患；
4. 避险场所：状态、容量、无障碍属性、来源和更新时间；
5. 路线：起点/终点、候选路线、风险证据、未知风险和失效时间；
6. 我的：订阅区域、语音设置、字号/对比度、报告记录和隐私说明。

位置权限拒绝时，用户仍可以搜索或选择区域。首页必须能在 mock 数据下脱离真实地图 key 演示。

## 九、后台必须交付的页面

最少实现：

1. 态势总览：辖区风险、预警、待核验报告、数据新鲜度；
2. 报告队列：查看、合并、核验、驳回、过期；
3. 地图管理：道路事件、风险点、避险场所；
4. 任务与通知：创建、发送、确认、失败、重试、升级；
5. 审计日志：按角色、资源、时间和 requestId 查询。

后台所有高风险动作都需要角色权限、二次确认和审计记录。

## 十一、AI 适配器要求

AI 只处理结构化、最小化和已授权的信息。先实现：

- 官方预警 -> 行动卡摘要；
- 多条报告 -> 去重/优先级建议；
- 社区事件 -> 交接班和复盘草稿；
- 结构化行动卡 -> 语音播报文本。

统一输出：

```json
{
  "summary": "...",
  "actions": ["..."],
  "evidence": [{"sourceId": "...", "observedAt": "..."}],
  "uncertainty": "...",
  "needsHumanReview": true,
  "generatedAt": "...",
  "expiresAt": "..."
}
```

对模型输出执行 JSON schema 校验、来源 ID 校验、过期检查和人工审核策略。用户报告中的 prompt injection、伪造官方口吻、要求改变权限的内容都只能作为普通文本处理。

## 十二、通知和降级

先实现 `MockNotificationProvider`，支持：

- queued、sent、delivered、acknowledged、failed、retrying；
- 幂等键，避免重复发送；
- 失败重试和超时升级；
- 测试中可查询完整投递记录。

真实短信、电话、微信订阅消息、App push 只能通过 provider 接入。没有凭证时，所有发送动作只写入 mock outbox，禁止真实外发。

关键降级：

- AI 失败：模板行动卡；
- 天气源失败：显示最近数据和过期标记；
- 地图源失败：显示避险场所和官方入口，不生成新路线；
- 数据库只读或高峰：优先返回缓存的附近摘要；
- 定位拒绝：搜索/选区；
- 无网络：显示缓存内容和缓存时间。

## 十三、开发阶段

### Phase 0：仓库和可运行骨架

- 检查仓库并创建/更新 `CLAUDE.md`；
- 初始化或识别现有前端、后端、数据库结构；
- 创建 `.env.example`、Docker Compose 和 mock 模式；
- 加入健康检查、统一错误响应、requestId 和基础日志；
- `README` 能让新环境启动项目。

验收：开发者能启动 API、数据库和至少一个前端；健康检查通过；没有真实外部调用。

### Phase 1：领域模型、迁移和风险引擎

- 创建 PostGIS 迁移；
- 建立官方预警、观测、报告、道路事件、避险场所、风险快照、审计日志；
- 创建 seed fixtures；
- 实现确定性风险计算和 freshness/confidence；
- 为正常、过期、冲突、缺失、越界输入写测试。

验收：可以通过 fixture 回放一个事件，输出稳定且可解释的风险快照。

### Phase 2：居民端闭环

- 实现首页、地图图层、上报、场所、路线、我的；
- 位置权限可选；
- UI 支持老年模式；
- 接入 mock API；
- 增加空态、加载态、过期态、无网络态和错误态。

验收：不配置地图 key 也能完成演示；用户可以完成“查看风险 -> 上报 -> 查看避险场所 -> 获取带证据路线”。

### Phase 3：社区/应急后台

- 实现态势总览、报告审核、道路和场所管理、任务通知、审计；
- 实现角色权限和高风险操作二次确认；
- 通过 mock provider 演示通知闭环。

验收：居民提交报告后，后台能核验并影响风险/路线；所有操作有审计。

### Phase 4：真实数据适配器

- 先读取正式接口文档和授权边界；
- 为每个数据源实现独立 adapter；
- 添加字段校验、限流、重试、时间戳检查、空间范围检查；
- 真实数据和 mock 数据都保留可回放 fixture；
- API key 只从环境变量读取，不能进入仓库和日志。

验收：真实源故障、超时、字段变化和撤销事件都能触发降级和告警。

### Phase 5：AI 和语音

- 实现 Noop/Mock AI；
- 增加结构化输出校验、来源校验、人工审核状态；
- 接入模板语音或平台 TTS；
- 增加 prompt injection、无证据和过期数据测试。

验收：AI 关停时核心功能仍可用；所有 AI 文案能回溯到来源。

### Phase 6：演练和上线前加固

- 执行对抗性测试矩阵；
- 测试高峰、断网、数据源冲突、通知重复、权限越权、恶意上传；
- 进行一次桌面推演；
- 生成演练报告和待改进清单；
- 只在用户明确授权后部署或接入真实通知渠道。

## 十四、强制测试矩阵

至少覆盖：

### 数据和时间

- 预警过期、撤销、重复、时间倒退；
- 不同来源冲突；
- 缺失坐标、非法几何、未来时间戳；
- 时区和夏令时边界；
- 旧报告不能覆盖新报告。

### 风险和路线

- 无数据不得输出绿色安全状态；
- 官方封路必须硬阻断；
- 只存在未知风险时不能标记安全；
- 无可用路线时返回明确失败原因；
- 路线结果到期后前端显示过期。

### 权限和隐私

- 居民不能核验或修改报告；
- 社区不能读取其他辖区的精确位置；
- 未授权角色不能发送通知或修改规则；
- 位置拒绝后核心公共功能仍可用；
- 精确位置不能出现在日志、公开 API 或 AI 请求中。

### 内容和 AI

- 报告文本包含“忽略系统规则”等 prompt injection；
- 报告伪造官方身份或发布命令；
- AI 无证据、证据过期、来源冲突；
- 模型返回非法 JSON、超长文本或危险承诺；
- AI provider 超时和异常。

### 可靠性

- 通知幂等、失败重试、超时升级；
- 地图/天气/AI 服务分别故障；
- 缓存显示最近更新时间；
- 数据库短暂不可写时不丢失用户报告；
- 上传大小、格式、恶意文件名和病毒扫描接口边界。

## 十五、每个阶段的输出格式

完成一个阶段后，用以下格式报告：

1. 本阶段已实现的用户可见结果；
2. 修改的文件；
3. 执行的命令和真实结果；
4. 已覆盖的边界/对抗性测试；
5. 仍未验证的内容及原因；
6. 下一阶段唯一最重要的工作。

不要把“代码已写入”当成“功能已完成”。只有运行结果支持的内容才能标记为完成。

开始 Phase 3。实现社区/应急后台：辖区态势、报告核验、道路事件、避险场所、任务通知、投递状态和审计。增加 RBAC、越权测试、二次确认和幂等通知。居民报告只有核验后才能成为高可信证据。完成后验证“居民上报 -> 后台核验 -> 风险/路线更新 -> 任务闭环”。
```

### Phase 4 续接

```text
开始 Phase 4。先整理正式数据源的 adapter 接口和授权/字段映射文档，再实现一个真实源适配器。没有正式接口和凭证时只实现 adapter、fixture、校验、重试、过期和故障降级，不得猜测 API。验证超时、字段缺失、重复事件、撤销事件、空间范围异常和密钥不进入日志。
```

### Phase 5 续接

```text
开始 Phase 5。实现 Noop/Mock AI、结构化输出校验、证据绑定、人工审核状态和模板语音。使用报告文本测试 prompt injection、伪造官方身份、无来源结论、过期证据和非法 JSON。关闭 AI provider 后，风险、预警、上报、后台和路线核心功能必须继续工作。
```

### Phase 6 续接

```text
开始 Phase 6。进行上线前对抗性审查。重点检查：任何未知风险是否被错误显示为安全；官方封路是否真的阻断；位置是否进入公开响应或日志；角色是否越权；通知是否重复发送；AI 是否生成无来源的精确数值或救援承诺；断网和数据源故障是否有明确降级。修复发现的问题，运行完整检查，生成演练报告和未验证项清单。不要部署生产或发送真实通知。
```

## Claude Code 项目级 `CLAUDE.md` 建议内容

项目初始化后，可要求 Claude Code 将以下内容写入根目录 `CLAUDE.md`，并根据实际命令补全：

```text
# FloodShield 项目约定

## 先读文件

- FLOOD_PLATFORM_SPEC.md
- docs/architecture.md
- docs/ai-safety.md
- 当前包管理器和测试配置

## 关键安全规则

- 官方预警、平台风险等级和 AI 摘要必须分开。
- 缺失/过期数据不能默认为安全。
- 路线必须返回证据、更新时间和失效时间。
- 用户报告先待核验，核验后才能升级可信度。
- 不保存后台持续定位，不把精确位置放入公开 API 或日志。
- mock provider 默认开启，真实外部发送必须显式配置和人工确认。

## 开发习惯

- 先读代码和测试，再做最小改动。
- 每个安全关键分支都要有测试。
- API 改动同步更新 OpenAPI、前端类型和测试。
- 保留用户已有修改，禁止使用 destructive git 命令。
- 每次完成必须报告命令和结果，未验证内容单独列出。
```
